      Recommendation as Generation: Unifying Personalized Video Generation and Recommendation at Industrial Scale

      推荐即生成：工业规模上统一个性化视频生成与推荐

      🧭 生成式推荐 · Semantic ID · 多智能体视频生成 ★ 深度解读

      Kuaishou Technology · Beihang University

      提出 RaG 范式，以解耦语义 ID 连接生成式推荐与多智能体视频生成，在快手广告场景实现闭环优化并获得广告收入提升

      📌 工作参考价值（4 点）
      • 解耦语义 ID 作为推荐与生成的统一接口：将视频表征拆解为内容语义与创意风格两部分 ID，既用于用户兴趣建模，又直接驱动下游视频生成。电商推荐中，同样可以设计分离的语义码本，分别控制商品推荐逻辑与展示素材风格，实现从“选品”到“创品”的链路打通。
      • 多智能体分层视频生成架构：用三个角色特化的 Agent（视觉规划、音频对齐、效果增强）顺序执行，共享一个 LLM 骨干并重用 KV-cache。在电商广告素材自动生成中，可借鉴 Agent 协作流程和 append-only 状态序列化的 KV-cache 复用，降低大批量素材生成的推理延迟。
      • 带约束的多目标奖励优化（GDPO + PID Lagrangian）：将用户反馈作为主目标，内容对齐与视频质量作为约束，通过组内奖励归一化和 PID 控制的拉格朗日乘子更新，稳定优化三个异构奖励信号。适合多目标推荐场景，如在优化点击率时控制素材质量与相关性阈值。
      • 近线生成 + 缓存策略解耦实时推理：视频生成延迟高，通过 SID 索引缓存已生成视频，在缓存命中时几乎零延迟返回，未命中时用最近邻视频占位并异步补全。对于个性化落地页、动态广告素材等非实时性要求稍低的场景，可采用“近线生成 + 分层缓存”架构，平衡生成质量与响应时间。

      TokenMinds: Pretrained User Tokens and Embeddings for User Understanding in Large Recommender Systems

      TokenMinds：基于语义ID与LLM的工业级双模态用户建模

      🧭 生成式用户建模 · Semantic ID ★ 深度解读

      Google DeepMind · YouTube

      提出双输出编码器-解码器架构，同时生成离散语义ID用户令牌和稠密嵌入，在YouTube排名系统中实现互补增益

      📌 工作参考价值（5 点）
      • 离散用户表示对齐候选集：将用户兴趣压缩为与物品共享的SID前缀，可用Prefix Embedding Mapping或可学习子词嵌入（LE）投影到稠密空间。电商场景可对商品SID做同样设计，直接用SID序列表示用户历史与意图，减少跨模态鸿沟，尤其适合图文、搜索词等多模态融合。
      • 双输出设计平滑过渡：同时产出稠密嵌入和离散令牌，既兼容现有DNN排序模型，又注入离散语义信号。在现有推荐系统引入生成式用户建模时，可采用类似双通道策略，避免全量重构；稠密侧做主力特征，离散侧做互补特征，通过注意力池化等方式融合。
      • 异步推理架构解耦成本：在用户行为服务（UBS）框架下离线生成用户表征并缓存，实时打分仅读取，将重生成延迟完全屏蔽。大模型上线时，可参照此模式以KV缓存和定时刷新实现超高吞吐，单用户推理339ms在后台完成，存储仅需1280B，这对电商/广告海量用户场景极具参考价值。
      • 跨场景统一建模提效：通过<LFV>/<SFV>等场景条件令牌，以及多上下文解码（共享编码器一次前传、分支波束搜索），将长短期视频场景合入单模型，训练算力减半，推理算力降低31%。若业务存在多个推荐流（如猜你喜欢、购后推荐、首页Feeds），可用相同技术方案统一建模，降低维护成本并从异质信号迁移学习。
      • Token适配与训练技巧：训练时采用前瞻窗口采样（预测未来24h内多个目标，非下一项）、截断SID前缀（更粗粒度）、历史长度约1K watch即可饱和，这些都是可迁移的调参经验。可学习嵌入（N-gram/SPM）远优于静态前缀映射，下游模型重新学习SID语义空间能带来显著线上收益。

      Do Generative Recommenders Deepen the Information Cocoon? A Closed-Loop Simulation with LLM-powered User Simulators

      生成式推荐会加深信息茧房吗？一项基于LLM用户模拟的闭环仿真研究

      🧭 生成式推荐信息茧房闭环模拟 ★ 深度解读

      Shandong University · Renmin University of China · Shandong Provincial Key Laboratory of Computing-Network Integration · Qingdao Key Laboratory of Trustworthy Artificial Intelligence

      生成式推荐减缓类别级曝光塌缩但仍在代码空间形成结构性茧房，语义ID与更大模型有助于抵抗蚕食

      📌 工作参考价值（4 点）
      • 用LLM用户模拟器做闭环评估：在生成式推荐上线前，可参考RecLoop框架搭建“推荐-交互-重训”循环，用LLM agent模拟用户偏好演化，评估长期信息茧房风险，尤其适合需要对算法负向效应进行离线预测的电商/广告场景。
      • 监控代码空间的结构化茧房：除了曝光指标，还应监控生成式模型离散码空间的层级熵与Top-κ集中度，重点观察粗粒度层的塌缩，一旦发现早期路由码高度集中，即使终端物品多样化，信息茧房已开始形成。

      • 模型容量作为缓冲：更大规模的生成模型（如OneRec 3B vs 0.5B）能保持更宽广的活跃码空间，对抗茧房塌缩，所以在算力允许时选择较大LLM backbone有助于长期信息多样化。

      📌 工作参考价值（4 点）

      OneRetrieval: Unifying Multi-Branch E-commerce Retrieval with an Editable Generative Model

      🧭 生成式检索 · 可编辑Semantic ID ★ 深度解读

      Kuaishou Technology

      首个兼具高召回与倒排索引级实时编辑能力的生成式检索框架，通过关键词对齐编码和预留槽位实现多路召回统一。

      📌 工作参考价值（4 点）
      • 保留槽位实现术语实时注入：在码本中为每个属性组预留少量 token，训练时不绑定具体词，仅通过自路由监督保证可解码；部署后只需更新属性词典和 SID-to-item 查找表即可立即上线新词，无需重训模型，完美替代倒排索引的编辑能力。
      • 关键词对齐编码 (KAE) 解耦语义与编辑：将语义标识符的每个位置绑定到可解释的属性词而不是量化嵌入，使编辑能力来源于上游字典，与模型策略分离；同时利用 LM 先验提升召回，位置透明便于 debug 和人工干预。
      • 信息论驱动的非均匀容量分配：通过互信息损失聚类将 18 类细粒度属性合并为 6 组，密集组分配更大码本容量（2048 vs 1024），显著减少属性词冲突，提升深层召回（HR@350 达 0.6055）且几乎不增推理成本。
      • 四阶段 SFT 联合保障质量与可编辑性：阶段 0 对齐槽位与属性词先验，阶段 2 引入协同共现信号为预留槽路由打下基础，阶段 3 用极小量自路由样本保持预留槽可发射；新增功能与检索质量通过近解耦的 stage 分别保障，避免相互干扰，可直接迁移到其他生成式推荐系统。

      🧭 生成式推荐 · Semantic ID 诊断与剖析 ★ 深度解读

      Huawei Technologies

      提供一种映射层诊断资源，在生成式推荐训练前揭示 SID 分词器的地址空间缺陷。

      📌 工作参考价值（4 点）
      • 映射健康检查前置：在启动 generator 训练前，用利用率、别名、前缀邻域对齐、流行度分配、结构成本等探针快速诊断 SID 分词器产物，避免因覆盖缺失或严重别名导致 GPU 浪费。可封装为内部评估模块，集成到 pipeline 中。

      OneFeed: A Unified Generative Framework for Feed ContentEnhancement and Query Generation

      OneFeed：统一生成式框架同时建模商品推荐与搜索Query生成

      🧭 生成式推荐 · Semantic ID 联合 Query 生成 ★ 深度解读

      用一个共享编码器同时生成推荐用的语义ID和搜索用的自然语言Query，并通过对比对齐弥补语义鸿沟

      📌 工作参考价值（4 点）
      • 统一 Feed 与搜索的生成范式：共享行为编码器同时产生 Semantic ID 和自然语言 Query，可以借鉴到电商的「千人千面」Feed 流中，让一个模型既做推荐又做搜索类目/query 推送，减少维护两套系统的成本。
